III

The Ramp — graduated autonomy (D2)

per-tenant, per-action-category config setting from day one
LVL 0
off

Category disabled entirely for this tenant.

LVL 1
draft_approve

Agent drafts; a human approves, edits or rejects in the inbox. Every event lands in the ledger.

LVL 2
auto_send_with_undo

Sends on its own; reversible through the ledger's undo window.

LVL 3
autonomous

Full autonomy for the category, still fully logged.

Graduation is earned by datae.g. ≥95% of drafts sent unedited over several weeks — measured from the first draft ever shown.
Circuit breakerEdit-rate or negative-reply spike auto-demotes a category back to draft_approve (the outreach bounce-breaker pattern, reused).
Never graduatesAnything committing money, anything legal, anything the health classifier flagged. Regardless of stats.
IV

The Gates — blocking, external lead time

neither blocks starting modules 0–1
GATE G1 · blocks customer #2
Google OAuth → Production + CASA

External → Testing refresh tokens die every ~7 days — tolerable for customer #1, fatal for customer #2. Gmail is a restricted scope: CASA verification takes weeks-to-months and a few hundred–few thousand €/yr via an authorized lab.

Action: start now, independent of everything else. Longest external dependency on the platform.

GATE G2 · blocks support-automation ingestion
Health-data decision (HDS)

Support automation ingests client correspondence; for a health broker that's where questionnaires médicaux live. HDS scope would rule out plain Atlas + Vercel for that data. Working lean: contractual carve-out + ingestion-side quarantine.

Action: ask what a typical month of client email contains → decide exclusion vs HDS before a single client email is ingested.

The Law — standing rules

inherited from the brief; agentic modules will test every one
Config, not code

Now including prompts. Prompt templates live alongside FieldDefinition, seeded per vertical. The broker prompt pack is a sellable asset, not source.

Generic ontology, vertical labels

Data-model names stay generic (knowledge base, conversations, playbooks); French broker vocabulary lives in labels/config — the stage-definition pattern.

Router only

Tenant data only through the DB router — agent steps too. Never a hardcoded connection.

Don't break the live app

Customer #1 is in production. Every agentic feature ships behind per-tenant config, default off.

One bridge

Every side-effectful agent action goes through the ledger. No exceptions, no per-module forks.

Events from day one

Emit structured events immediately; build dashboards later. Events cannot be backfilled.

IX

The Toll — pricing model, by customer volume

for Nicolas · cost-plus at a medium-low margin, not a premium-SaaS margin

Same tiers as §VII — how many clients the customer manages — because that's also what drives Nicolas's cost to serve (AI runtime scales with support/RCA/lookup volume; Atlas, Vercel and the build subscription are near-fixed per tenant). "Medium-low margin" is read here as ~30% gross margin, well under the 70–85% a typical SaaS product runs — deliberately, to price for Path C's early adopters before volume justifies a premium. Recompute once real Vercel/Atlas/Claude invoices exist; the memo's cost snapshot (§V) is the source for the cost column.

Client-book tierCost to serve /moPrice /moGross marginCustomer's value /moCustomer ROI
Up to 5 clients€45€6531%€4176.4×

Reads as: €65–199/mo depending on tier, holding Nicolas's margin near 30% throughout, while the customer still sees a 5–6× return on the subscription in hours-saved value alone — before counting faster replies or deals that don't go cold. Margin compresses slightly at the low end because the €35–40 infra floor (Atlas + Vercel + amortized build subscription) doesn't shrink with a small client book — the same dynamic the memo flags for customer #1 vs #2 in reverse: small tenants are the least profitable per euro, large ones the most, so the real margin lever is total tenant count (§V), not any single customer's tier.

Two levers to raise the margin later without raising it above "medium-low" prematurely: Batch API + prompt caching (already assumed in the cost column, −50%/−90% per the memo) and amortizing the €35 infra floor across more tenants as the book grows past customer #2–3.
